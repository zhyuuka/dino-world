# 游戏开发项目规则

## 〇、最高原则：质量优先
- **本项目以"最终成品质量"为第一优先级**，所有技术选型、配置、代码实现都必须以产出高质量可运行游戏为准。
- 用户只关心最终结果（能玩的游戏），不关心中间代码是否易读。
- 当"易学易读"与"质量性能"冲突时，**一律选质量性能**。
- 引擎、语言、库的选型以"能产出最佳质量成品"为标准，不以"小白能看懂"为标准。
- 即便如此，对用户的面沟通解释仍用日常语言，不丢术语黑话。

## 一、面向用户约定
- 本项目的最终用户（甲方）是**完全不懂任何软件知识的小白**。
- 所有面向用户的解释必须使用**日常语言**，避免技术黑话；不可避免时必须配通俗类比。
- 用户只想要结果，不要把代码细节、中间日志当作交付物抛给用户。
- 内部技术实现可以复杂、专业、用强类型/工程化方案，但**对外只报告"能玩到什么"**。

## 二、项目性质硬性约束
- **本项目必须是一个原生/独立游戏软件（如 .exe / .app / APK 等可运行程序），严禁做成 HTML 网页游戏。**
- 不允许用纯 HTML+JS 单文件、浏览器内运行的方式作为最终交付物。
- 最终交付物应能在目标平台上脱离开发环境独立运行。
- 当前项目：3D 开放世界恐龙生存游戏，目标平台 Android，最终交付 APK。

## 三、配置责任划分
- **所有开发环境的配置（引擎版本、语言、构建工具、依赖、目录结构、IDE 等）均由 AI 助手自行决定与配置**。
- AI 助手对所选配置的稳定性、可运行性、产出质量负责。
- 配置选择遵循"质量优先"原则，不为了简单而牺牲性能/可维护性。

## 四、开发前必做：需求澄清
- **在写任何一行游戏代码之前，必须先向用户问清所有游戏细节和配置需求。**
- 必问维度至少包含：游戏类型/玩法、2D/3D、目标平台、主题世界观、单人/多人、美术风格、体量、附加系统。
- 每次提问使用 AskUserQuestion 工具，单次最多 4 个问题，分批问完所有维度。
- 对不切实际的需求（如新手要做开放世界+联机），必须诚实告知并给出分阶段方案。

## 五、技术选项推荐与说明规范
- 涉及**参数、引擎、语言、库、构建方式**等技术选项时，AI 必须给出**推荐选项**，并对每个选项标注其**效果/含义**。
- 推荐选项放第一位，标注"（推荐）"。
- 每个选项下方用一句话写清"选了会怎样、不选会怎样、对最终成品意味着什么"。
- 禁止只丢术语不给解释；禁止让小白在没有任何说明的情况下做技术决策。

## 六、问题处理原则
- **遇到任何报错、不熟悉的 API、版本兼容、构建失败等问题，必须优先使用 WebSearch 工具联网搜索最新资料，而不是依靠训练知识库死磕。**
- 搜索时使用当前年份（2026）以确保资料时效性。
- 知识库信息仅作为辅助，与官方文档冲突时以官方文档为准。
- 搜索后仍无法解决时，再向用户说明并请求决策，不得无限重试同一个失败操作。

## 七、云端协作与多会话同步
- 开发在 TRAE 云端环境进行，必须使用 **GitHub 仓库**做跨会话代码同步（当前为 Public 公开仓库）。
- 每个会话开始：`git pull` 拉取最新；每个会话结束前：`git commit && git push`。
- 主会话上下文宝贵（200k），**重度实现任务（写代码、调报错、搭场景）一律委托给 Task 工具的 general_purpose_task 子 Agent**。
- 主会话只负责：需求对接、决策、向用户汇报、子 Agent 调度。
- 项目进度、阶段里程碑、未决事项必须写入仓库内的 `PROGRESS.md`，每个会话开始时主 Agent 读取它来恢复上下文。

## 八、开发流程顺序
1. 写入并确认项目规则（本文件）
2. 通过 AskUserQuestion 完成需求澄清
3. 基于需求确定技术栈，向用户公示选型与理由
4. 搭建项目骨架（引擎初始化、目录、依赖、构建脚本）
5. 分模块实现游戏功能，每完成一块即向用户演示/说明
6. 打包为目标平台可运行程序并交付

## 九、分阶段开发计划（恐龙开放世界项目专属）
项目分 4 阶段推进，每阶段都是可玩成品：
- **第 1 阶段（当前）**：小型 3D 场景 + 1 只玩家恐龙（走/跑/咬/视角控制）+ 几只 AI 恐龙。**单机**。
- **第 2 阶段**：吃/饥饿/血量/成长进化系统。
- **第 3 阶段**：扩大地图、加多种恐龙、加探索/资源点。
- **第 4 阶段**：多人联机（如条件允许）。

每阶段做完一阶段再谈下一阶段，禁止跨阶段超前实现。

## 十、交付与验证
- 每个里程碑都要给出"用户能直接运行/看到的东西"。
- 避免长篇理论，能用截图、运行结果说话就不要只用文字。
- 中间产物（编译日志、依赖冲突等）由 AI 自行消化，只向用户报告结论。
- 用户最终拿到的是 APK，能装到安卓手机上玩。
